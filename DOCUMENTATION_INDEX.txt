🔧 FOR IMPLEMENTING ALL-TEAM REPORT FEATURE
═══════════════════════════════════════════════════════════════════════════════

START HERE:
  1. Read QUICK_REFERENCE.md Section "FOR ADDING ALL-TEAM REPORT FEATURE"
  2. Read FLOW_ANALYSIS_PART2.md Section 10 (detailed breakdown)

KEY INSIGHTS:
  • FORMAT B emails already contain multiple teams
  • _processSinglePairing() does most of the work per team
  • reportBuilder is embed-agnostic, can be reused
  • Need to aggregate pairings, not just send individually

APPROACH OPTIONS:
  1. Command-driven: @Clerk Kent all-team report [round]
  2. Email-driven: Detect FORMAT B, send consolidated report
  3. Batch-mode: Collect all pairings, send one message

REUSABLE CODE:
  • Line 565-614 in bot.js (opponent lookup logic)
  • Line 616-649 in bot.js (judge lookup logic)
  • All of report-builder.js (embed building)
  • caselist, paradigm, notion services (lookups)

NEW CODE NEEDED:
  • Aggregator function to collect team pairings
  • New embed format for multiple teams (table? grouped?)
  • Channel selection for consolidated report
  • Potentially: dedupe logic for multi-round emails

STORAGE CHANGES NEEDED:
  • Track when all-team report was sent (prevent duplicates)
  • Maybe: track "current round" for command-driven reports

📝 NOTES FOR IMPLEMENTATION
═══════════════════════════════════════════════════════════════════════════════

STRENGTHS OF CURRENT DESIGN:
  ✓ Clear separation: setup → confirmation → processing
  ✓ Deduplication built in (processedEmailUids)
  ✓ Flexible channel mapping (auto + manual overrides)
  ✓ Services layer (caselist, paradigm, notion) isolated
  ✓ Embed building separate from data collection
  ✓ Per-team processing loop easy to extend

FOR ALL-TEAM REPORT:
  ✓ Can reuse _processSinglePairing() helper logic
  ✓ Can reuse reportBuilder methods
  ✓ activeSession.allEntries already has all opponents
  ✓ activeSession.channelMappings has all team channels

POTENTIAL CHALLENGES:
  ⚠ Rate limiting on paradigm/caselist lookups (many teams at once)
  ⚠ Embed size limits (10 max per message)
  ⚠ Message length limits (if aggregating all info)
  ⚠ Deduplication logic (send once per round, not multiple times)
